# PRIZM 4.0 — Baseline audit, 18 May 2026

The first end-to-end audit of PRIZM. Captures the system's state on the date above so future audits have something to compare against. Findings are grouped by severity, then by category. Each finding has a status field that gets updated as it's resolved.

## How to use this document

- **The findings list is the active tracking list** until something more durable replaces it. When a finding is resolved, mark it `RESOLVED` with a date and one-line note, but don't delete it — the trail matters.
- **The proposed flow** at the bottom describes the cadence for resolving findings. Follow it unless we explicitly agree otherwise.
- **Decisions needed** are flagged inline and collected at the bottom. Don't try to "just fix" something flagged as needing a decision — surface the decision to Alvin first.
- **Caveats on methodology** are noted where relevant. The contrast math in particular should be cross-checked with a dedicated tool before any token values change.

## Scope of the audit

Read-only checks against the working tree at commit `6e75fde` (HEAD on `main`, 18 May 2026):

1. **Code-health audit** — TODO / FIXME comments, type-safety holes (`any`, `@ts-ignore`), `console.*` leftovers, lint-rule suppressions, raw Tailwind colour utilities in components, dependency staleness, CVEs

3. **API drift audit** — `components-registry.ts` slugs vs `components-api.ts` keys
4. **Documented-but-not-wired audit** — motion tokens (`--prizm-duration-*`, `--prizm-ease-*`) usage
5. **Stale-claim audit** — assertions in `AI_HANDOFF.md` that are no longer true

NOT in scope (intentional — separate later passes):
- Accessibility (keyboard sweep, screen-reader smoke test, axe-core)
- Visual / design audit (taste judgements on spacing, typography, theme coherence)
- Cross-browser testing
- Performance / bundle size
- Production deploy verification
- Air-gap tarball cold-start verification

## Findings

### Red — must address

#### R1. Contrast failures below WCAG AA in multiple themes

**Status**: RESOLVED 2026-05-19 per D1 Path A + D2 Option 2. Token changes applied across all four theme variants; copy softened on `/c3/overview` from "AAA where practical" to "AA across all four themes, AAA for primary text where the design naturally allows." See D1 / D2 sections below for the agreed policy and the exception note for `danger-fg on danger` in dark modes.

**Methodology caveat first**: contrast ratios computed by converting `oklch(L% c h)` token values through the standard Oklab → linear sRGB matrix to relative luminance, then applying the WCAG contrast formula. The conversion is correct in principle but my output was not cross-checked against a reference tool. Treat exact ratios as ±0.2; the *direction* of these findings should be reliable.

**c3-light** (5 failures or near-failures against AA 4.5:1 for normal text):
- `success` on `bg` — **2.99** (green-500 on white)
- `warning` on `bg` — **2.96** (amber-500 on white)
- `info` on `bg` — **3.73**
- `danger` on `bg` — **4.42**
- `fg-subtle` on `bg` — **4.42**

**enterprise-light** (3 failures):
- `success` on `bg` — **3.08**
- `warning` on `bg` — **2.06** (worst in the whole system — amber-500 on white)
- `info` on `bg` — **3.85**

**c3-dark** (3 failures):
- `fg-subtle` on `bg` — **4.33**
- `fg-subtle` on `surface` — **3.74**
- `danger-fg` on `danger` — **3.66** (the close button on a red badge / banner — practical problem)

**enterprise-dark** (3 failures):
- `fg-subtle` on `bg` — **4.23**
- `fg-subtle` on `surface` — **3.74**
- `danger-fg` on `danger` — **3.66**

**Patterns**:
- Status hues (success / warning / info) used *as text* fail in light modes because the 500-step on white is too light. Two solutions possible — see D1.
- `fg-subtle` on `bg` / `surface` fails in three of four themes by design (tertiary text). But the C3 copy promises "WCAG AAA where practical"; this promise is broken. See D2.
- `danger-fg on danger` failing in both dark modes is the most pressing single failure because the pairing actually appears in real components (close button on a destructive badge / toast).

**Many other pairs pass AA but fail AAA** (between 4.5 and 7). If the AAA-where-practical promise is held, those are also in scope.

#### R2. Security vulnerability — postcss XSS via Next 15.5.18

**Status**: RESOLVED 2026-05-18 via `pnpm.overrides` in `package.json` forcing `postcss@^8.5.10`. `pnpm audit` now reports "No known vulnerabilities found".

`pnpm audit` reports one moderate vulnerability: GHSA-qx2v-qp2m-jg93, XSS via unescaped `</style>` in postcss's CSS Stringify Output. Affects `postcss < 8.5.10`. Pulled in transitively as `next@15.5.18 > postcss@8.4.31`.

**Resolution options**:

- Override the postcss version via `pnpm.overrides` in `package.json` to force `^8.5.10`

Either fix is low-risk. Recommend the override route since it's transparent and doesn't tie us to a specific Next patch release.

## Decisions needed

These cannot be resolved without input from Alvin. Don't proceed past `R1` or `Y2` or `Y3` without each respective decision.

### D1. Status hues — fix the tokens, or document them as fill-only?

**Decision 2026-05-19: Path A — bump tokens darker in light modes.**

Light themes (c3-light + enterprise-light): `success`, `warning`, `info`, `danger` all reference the new `700`-step status hues in `baseline.css`. Dark themes stay at 500 (passes 3:1 as fill and 4.5:1 as text on dark bg). `c3-light` also bumps `fg-subtle` from slate-500 → slate-600 to pass AA on the slightly-darker `bg`. `enterprise-dark` bumps `fg-muted` slate-400 → slate-300 because the new `fg-subtle` would have collided with it.

Rejected Path B (`*-fg` partners) because the suffix would have meant something completely different from existing `accent-fg` / `danger-fg` (which mean "foreground that sits on saturated fill"), and the naming clash would confuse component authors and AI consumers. Rejected Path C (`-strong` variants) as too much token surface for the problem.

### D2. The "AAA where practical" promise — hold or soften?

**Decision 2026-05-19: Option 2 — AA as the hard floor, AAA where it naturally falls out.**

WCAG text-contrast threshold (4.5:1) applies to all clearly-text token pairings: `fg`, `fg-muted`, `fg-subtle`, `accent` (used as link text), `accent-fg` (text on accent fill), `danger-fg` (text on danger fill). Status hues (`success`, `warning`, `info`, `danger`) used as fills on `bg` only need WCAG's non-text-contrast threshold (3:1), which all four themes pass after the D1 token changes.

**One documented exception**: `danger-fg on danger` in c3-dark and enterprise-dark sits at ~4.56 with the new red-600 — passing AA but with little headroom. The fundamental tension is that white-on-saturated-red cannot reach AA contrast for red lighter than ~oklch(50% L); we chose the smallest-darkening red (600) that lets the pair pass. Component authors using `bg-danger text-danger-fg` for destructive buttons / close icons in dark mode should not assume large amounts of comfort margin and should reinforce the affordance with focus / hover states.

Rejected Option 1 (strict AAA) because it would force status hues to the 800-step in light modes, making them visually muted in a way that conflicts with the system's intent for "warning" and "success" to feel vibrant. Rejected Option 3 (zone-split AAA / AA) because the same primitive renders in both zones — per-zone conformance bars on the same component get confusing.

### D3. Motion tokens — wire or drop?

Currently five orphan tokens. Either:
- **Wire them through** — components consume `var(--prizm-duration-*)` and `var(--prizm-ease-*)` instead of Tailwind defaults. Animation behaviour changes visibly across the system. Need to also expose them as Tailwind utilities (`duration-fast`, `ease-out`) for consumer ergonomics.
- **Drop them** — remove from `baseline.css`, retire the `/docs/surface-and-motion` motion section, accept that motion is "use Tailwind defaults". Smaller token surface.

### D4. Dependency strategy

Two axes:
- **How far back to fall behind before we upgrade.** Major-version policy? One major behind is fine, two requires a deliberate upgrade plan?
- **Base UI version lock.** Pin `1.0.0-rc.0` until GA, track rc updates aggressively, or wait for GA and ride the release? Each has different breakage windows.

Both decisions feed into what `audit-deps.ts` should fail on.

## Proposed flow for resolving the audit

The principle: **fix-then-automate per category**. CI scripts encode the policy we want from this point forward, never the broken state we're recovering from.

### Phase 1 — Land the "no-decision-needed" fixes (one session, ~1 hour)

These have clear resolutions; no design call needed. Do all four in one commit.

- [x] **R2** — RESOLVED 2026-05-18. `pnpm.overrides` forces `postcss@^8.5.10`; `pnpm audit` clean.
- [x] **Y1** — WONT-FIX 2026-05-18 (audit was wrong; the `console.log` calls live inside a code-snippet template literal, not executing JSX).
- [x] **S1** + **S2** — RESOLVED 2026-05-18. Stale claims removed from `AI_HANDOFF.md`.
- [x] **`scripts/audit-components.ts`** added 2026-05-18 and wired into CI as `pnpm audit:components`. Verifies (1) every registry slug has a COMPONENT_API entry, (2) every COMPONENT_API key is in the registry, (3) every COMPONENT_API entry has at least one prop or sub-component. The third check caught two false-positive "stubs" during development (Toast, ContextMenu both have zero root props but rich sub-components) and the logic was tightened to `props + subComponents === 0` accordingly. Lesson: scripts catch what eyeballs miss; also catch where eyeballs got it wrong.

### Phase 2 — Resolve D1 + D2, then fix contrast and automate (one session for decisions, one for execution)

This is the biggest single category by surface area.

- [x] **D1 + D2 decided** 2026-05-19 — see decision sections above.
- [x] **Execution** 2026-05-19 — token changes applied to `baseline.css` (new 700-step status hues), `c3-light.css`, `c3-dark.css`, `enterprise-light.css`, `enterprise-dark.css`, plus the duplicated values in `app/globals.css` swatch classes. - [ ] **`scripts/audit-contrast.ts`** — pending. Will encode the text-vs-fill threshold split per D2, fail CI on violations, and be cross-checked against an external tool once. Coming as commit #2 of Phase 2.

### Phase 5 — Close the audit

When all findings are RESOLVED:
- Add a "RESOLVED — 2026-XX-XX" stamp at the top of this file
- Update `AI_HANDOFF.md`'s "Path to production-ready" section to remove this audit's findings from the blocking list
- Begin the next audit (`audits/2026-XX-XX-something.md`)

The audit doc itself is never edited to *remove* findings — every finding stays visible, just with status updated. This builds the historical trail.

## Conventions for future audits

- One audit doc per pass, dated YYYY-MM-DD
- Findings have a stable ID (R1, R2, Y1, etc.) so they can be referenced from commits, PRs, and other audit docs
- Status field on every finding: OPEN, RESOLVED, WONT-FIX (with reason)
- Methodology caveats noted inline where relevant — every audit involves some judgement, surface it
- "Decisions needed" is a real section, not a placeholder — if a finding can't move without a decision, flag it explicitly
- Each finding maps to an actionable next step
- No fixes are made *in* the audit doc — the doc records, work happens elsewhere

The point of the audit doc is that a future session (or a future audit) can pick up the trail without having to redo the work.
